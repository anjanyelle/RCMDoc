Healthcare RCM Application - MVP Guide (Part 2: Modules & Features)

Version: 1.0

For: Development Team & Technical Lead

3. Modules Included in MVP

Module 1: Appointment Scheduling

Purpose:

Schedule patient appointments and link them to encounters for billing.

What User Does:

  • Opens calendar view
  • Selects date and time slot
  • Selects patient and provider
  • Selects appointment type (New Patient, Follow-up, etc.)
  • Adds notes if needed
  • Clicks "Schedule"

    • What Backend Does:

  • Checks provider availability
  • Checks for scheduling conflicts
  • Creates appointment record
  • Sends confirmation (optional: via Twilio SMS)
  • Creates placeholder encounter

    • APIs Needed:

  • Internal: Appointment CRUD endpoints
  • Twilio API (optional for SMS reminders)

    • Database Tables:

  • appointments
  • provider_schedules
  • appointment_types

    • Features:

  • Calendar view (day/week/month)
  • Color-coded by appointment type
  • Drag-and-drop rescheduling
  • Automated reminders (24 hours before)

    • Module 2: Patient Registration

    Purpose:

    Capture and store patient demographic and insurance information accurately.

    What User Does:

  • Clicks "New Patient" button
  • Fills in patient form:
  • Name, DOB, Gender, SSN
  • Address, Phone, Email
  • Emergency contact
  • Insurance information (primary & secondary)
  • Clicks "Save"

    • What Backend Does:

  • Validates all required fields
  • Checks for duplicate patients (by name + DOB)
  • Generates unique patient ID
  • Stores data in PostgreSQL database
  • Returns success message

    • APIs Needed:

  • Internal: Patient CRUD endpoints
  • External: None for basic registration

    • Database Tables:

  • patients (demographics)
  • patient_insurance (insurance details)
  • patient_contacts (emergency contacts)

    • Validation Rules:

  • SSN format: XXX-XX-XXXX
  • Phone: (XXX) XXX-XXXX
  • Email: valid email format
  • DOB: Cannot be future date
  • Insurance policy number: Required if insurance selected

    • Module 3: Insurance Verification

    Purpose:

    Verify patient insurance eligibility and benefits before appointment to avoid claim denials.

    What User Does:

  • Opens patient record
  • Clicks "Verify Insurance" button
  • System shows verification results in 10-30 seconds:
  • Coverage status (Active/Inactive)
  • Copay amount
  • Deductible (met/remaining)
  • Out-of-pocket max
  • Prior authorization requirements

    • What Backend Does:

  • Extracts patient insurance data
  • Creates EDI 270 transaction (eligibility inquiry)
  • Sends to Waystar API or Availity API
  • Receives EDI 271 response (eligibility response)
  • Parses response and stores in database
  • Caches result for 24 hours

    • APIs Needed:

  • Waystar API (preferred for multi-payer support)
  • Availity API (alternative, free for some payers)

    • API Integration Example:

    
# Waystar eligibility check
POST https://api.waystar.com/eligibility/v1/inquiries
Headers:
  Authorization: Bearer {api_key}
  Content-Type: application/json
Body:
{
  "patient": {
    "firstName": "John",
    "lastName": "Doe",
    "dateOfBirth": "1980-05-15",
    "memberId": "ABC123456789"
  },
  "payer": {
    "payerId": "00431",  # Aetna
    "payerName": "Aetna"
  },
  "provider": {
    "npi": "1234567890",
    "taxId": "12-3456789"
  },
  "serviceDate": "2026-05-20"
}
Response:
{
  "eligibilityStatus": "Active",
  "copay": 25.00,
  "deductible": {
    "annual": 1500.00,
    "met": 800.00,
    "remaining": 700.00
  },
  "outOfPocketMax": {
    "annual": 5000.00,
    "met": 1200.00,
    "remaining": 3800.00
  },
  "priorAuthRequired": false
}

    Business Rules:

  • Verify before every appointment
  • Cache results for 24 hours
  • If inactive → patient becomes self-pay
  • Alert user if prior auth required

    • Module 4: Prior Authorization / Referral Management

    Purpose:

    Obtain insurance company approval before expensive procedures, specialist consultations, surgeries, diagnostic tests, or inpatient admissions.

    What User Does:

  • Opens "Prior Authorization" module
  • Clicks "Create Authorization Request"
  • Selects patient and scheduled service/procedure
  • Enters authorization details (CPT/Procedure codes, ICD-10 diagnosis codes, ordering physician, requested service date, facility information)
  • Uploads supporting clinical documents (doctor notes, lab reports, imaging reports, referral forms)
  • Selects insurance payer
  • Submits authorization request
  • Monitors authorization status (Pending, Approved, Denied, Additional information requested)
  • Tracks authorization expiry dates
  • Communicates with payer if follow-up is required

    • What Backend Does:

  • Verifies whether authorization is required for selected procedure
  • Generates unique authorization request ID
  • Stores authorization request in database
  • Sends authorization request to payer system/API

    • APIs Needed:

  • Internal: Authorization CRUD APIs, Referral management APIs, Document upload APIs, Notification APIs, Status tracking APIs
  • External: Insurance payer authorization APIs, Clearinghouse APIs, EHR/EMR integration APIs, Fax/secure messaging APIs (for non-digital payers)

    • Database Tables:

  • prior_authorizations
  • authorization_status_history
  • authorization_documents
  • referrals
  • payer_communications
  • authorization_expiry_tracking

    • Business Rules:

  • Authorization Requirement Rules
  • Referral Management Rules
  • Clinical Documentation Rules
  • Authorization Submission Rules
  • Authorization Expiry Rules

    • Module 5: Patient Check-In

    Purpose:

    Confirm patient arrival, verify demographic and insurance information, collect copays, obtain consent forms, and manage patient queue flow before the medical visit.

    What User Does:

  • Opens "Patient Check-In" module
  • Searches patient by Appointment ID, Patient name, Phone number, or Patient ID
  • Verifies patient details (address, phone number, insurance information, emergency contact)
  • Updates any changed information
  • Reviews appointment details
  • Collects required digital consent forms (HIPAA consent, treatment consent, financial responsibility form)
  • Collects copay/payment from patient
  • Generates payment receipt
  • Marks patient as "Checked-In"
  • Updates patient queue/waiting status
  • Notifies medical staff about patient arrival

    • What Backend Does:

  • Retrieves scheduled appointment details
  • Validates appointment date and time
  • Verifies patient insurance eligibility
  • Checks outstanding balances or copay requirements
  • Stores updated demographic information
  • Saves signed digital consent forms securely
  • Processes payment transaction
  • Generates receipt and payment confirmation
  • Updates appointment status to Checked-In, Waiting, or Ready for Provider
  • Updates real-time patient queue
  • Sends notifications to nurse station, provider dashboard, and billing department
  • Logs complete audit trail of check-in activity

    • APIs Needed:

  • Internal: Appointment management APIs, Patient verification APIs, Check-in status APIs, Queue management APIs, Consent form APIs, Payment processing APIs, Notification APIs
  • External: Insurance eligibility verification APIs, Payment gateway APIs (Availity / Waystar), SMS/Email notification APIs (Twilio / SendGrid), Digital signature APIs

    • Database Tables:

  • patient_checkins
  • appointment_queue
  • patient_consent_forms
  • copay_transactions
  • checkin_audit_logs
  • appointment_status_history

    • Validation Rules:

  • Required consent forms must be signed before completion
  • Copay amount must match payer configuration
  • Payment transaction must succeed before finalizing check-in
  • Duplicate check-ins for same appointment are not allowed

    • Module 6: Clinical Documentation / EMR

    Purpose:

    Maintain a complete and structured Electronic Medical Record (EMR) containing all clinical information such as doctor notes, diagnoses, prescriptions, lab orders, and treatment plans for each patient encounter.

    What User Does:

  • Opens "EMR / Clinical Documentation" module
  • Selects patient from search or active encounter list
  • Creates or updates clinical notes (Chief complaint, History of present illness, Physical examination findings)
  • Adds diagnosis information (Primary diagnosis, Secondary diagnoses, ICD-10 codes)
  • Prescribes medications (Drug name, Dosage, Frequency, Duration)
  • Orders lab or radiology tests (Lab tests, Imaging like X-ray, CT, MRI)
  • Creates treatment plan (Procedures, Follow-up schedule, Care instructions)
  • Saves and signs the clinical documentation

    • What Backend Does:

  • Retrieves patient EMR history and active encounter data
  • Creates or updates encounter record in EMR system
  • Stores structured clinical data (Notes, Diagnoses, Prescriptions, Orders, Treatment plans)
  • Validates medical codes (ICD-10 for diagnoses, RxNorm/drug database for prescriptions)
  • Links clinical documentation to billing/RCM module
  • Triggers downstream workflows (Charge capture generation, Lab/radiology order processing)
  • Maintains version history of all clinical updates
  • Ensures compliance with HIPAA and audit logging
  • Locks signed notes from further edits (only addendum allowed)

    • APIs Needed:

  • EHR/EMR API (Epic / Cerner / FHIR APIs) – clinical data storage & retrieval
  • FHIR API Layer – standardized clinical data exchange
  • Internal Encounter API – visit/clinical note management

    • Clinical Validation Rules:

  • Encounter must be active before documentation
  • Diagnosis must use valid ICD-10 codes
  • Prescription must be validated against drug database
  • Controlled substances require additional verification
  • Lab/radiology orders must match clinical indication
  • Documentation must be signed before finalization
  • Signed notes cannot be edited (only amended via addendum)

    • Data Stored:

  • encounters
  • clinical_notes
  • diagnoses
  • prescriptions
  • lab_orders
  • radiology_orders
  • treatment_plans

    • Module 7: Medical Coding

    Purpose:

    Assign correct ICD-10 diagnosis codes and CPT procedure codes to encounters for accurate billing.

    What User Does:

  • Opens encounter after patient visit
  • Reviews doctor's notes
  • Searches and selects ICD-10 codes (diagnoses)
  • Searches and selects CPT codes (procedures)
  • Adds modifiers if needed
  • Links codes to encounter
  • Clicks "Save"

    • What Backend Does:

  • Validates code formats
  • Checks code validity (active codes only)
  • Validates code combinations (some codes can't be billed together)
  • Stores codes linked to encounter
  • Calculates expected reimbursement

    • APIs Needed:

  • Internal: ICD-10 and CPT code database
  • OpenAI API (optional for AI-assisted coding suggestions)
  • FHIR APIs (optional to pull codes from EHR)

    • AI-Assisted Coding (Optional MVP Feature):

    {"code": "99213", "description": "Office visit, established patient, moderate complexity", "confidence": 0.92}

    Database Tables:

  • icd10_codes (100,000+ diagnosis codes)
  • cpt_codes (10,000+ procedure codes)
  • encounter_diagnoses
  • encounter_procedures

    • Module 8: Charge Entry / Charge Capture

    Purpose:

    Capture and manage all billable services, procedures, medications, and provider charges accurately to support claim generation and revenue cycle workflows.

    What User Does:

  • Opens "Charge Entry / Charge Capture" module
  • Searches patient encounter or appointment
  • Selects provider and date of service
  • Enters billable items:
  • Procedures (CPT/HCPCS codes)
  • Medications
  • Supplies
  • Service charges
  • Applies modifiers if required
  • Reviews fee schedule amounts
  • Verifies diagnosis-to-procedure linkage
  • Saves charges for billing
  • Reviews charge validation alerts
  • Submits finalized charges for claim creation

    • What Backend Does:

  • Retrieves encounter and provider details
  • Validates CPT/HCPCS procedure codes
  • Validates diagnosis linkage using ICD-10 codes
  • Applies payer-specific fee schedules
  • Calculates charge amounts automatically
  • Validates modifiers and coding combinations
  • Detects duplicate or missing charges
  • Stores charge transactions in billing database
  • Generates audit trail for charge updates
  • Sends approved charges to claim generation workflow

    • APIs Needed:

  • Internal Charge Management API – charge entry & validation
  • Coding API (ICD-10 / CPT / HCPCS Lookup) – code verification
  • Fee Schedule API – payer pricing & reimbursement validation
  • EMR/EHR API (Epic / Cerner / FHIR APIs) – encounter & clinical data
  • Claim Generation API – billing workflow integration

    • Charge Validation Rules:

  • Every charge must be linked to a valid encounter
  • CPT/HCPCS codes must be active and billable
  • Diagnosis codes must support medical necessity
  • Required modifiers must be applied where necessary
  • Duplicate charges for same service are not allowed
  • Charge amount must match configured fee schedule
  • Provider must have valid credentials/NPI
  • Charges must be finalized before claim generation

    • Data Stored:

  • charge_entries
  • charge_details
  • provider_fee_schedules
  • charge_modifiers
  • charge_validation_logs
  • charge_audit_history

    • Reports Generated:

  • Charge Capture Summary Report
  • Missing Charges Report
  • Charge Validation Error Report
  • Provider Productivity Report
  • Fee Schedule Variance Report

    • Module 9: Claim Scrubbing

    Purpose:

    Perform pre-submission validation of healthcare claims to detect errors, missing data, and coding issues in order to reduce rejections and denials from payers.

    What User Does:

  • Opens "Claim Scrubbing" module
  • Selects claims in draft or ready status
  • Runs "Scrub Claims" process
  • Reviews scrubbing results (Errors, Warnings, Passed claims)
  • Opens each flagged claim
  • Fixes issues such as missing diagnosis or procedure codes, invalid CPT/ICD-10 codes, incorrect modifiers, missing provider or patient details
  • Re-runs scrubbing after corrections
  • Moves clean claims to "Ready for Submission"

    • What Backend Does:

  • Fetches claim data from billing database
  • Applies validation rules engine (required field checks, code format validation for ICD-10, CPT, NPI, medical necessity validation)
  • Cross-checks diagnosis vs procedure compatibility
  • Applies payer-specific rules and fee schedule consistency checks
  • Detects duplicate claims, missing modifiers, invalid or inactive codes
  • Generates scrubbing report (errors, warnings, suggested fixes)
  • Updates claim status (Scrubbed, Failed Scrub, Ready for Submission)
  • Stores scrubbing history for audit and analytics

    • APIs Needed:

  • Internal Claim Scrubbing Engine API
  • Waystar API (Claim Validation Rules Engine)
  • Availity API (Payer Rule Validation)
  • Coding Reference API (ICD-10 / CPT Lookup)
  • Fee Schedule API (Payer Contract Validation)

    • Validation Rules:

  • All claims must have valid ICD-10 diagnosis codes
  • CPT codes must match approved procedures
  • NPI numbers must be active and valid
  • Required modifiers must be present where applicable
  • No duplicate claims for same encounter
  • Diagnosis must support medical necessity for procedures
  • Charges must align with payer fee schedules
  • All required demographic and insurance data must be complete

    • Data Stored:

  • claims
  • claim_scrub_results
  • claim_errors
  • claim_warnings
  • scrubbing_rules_log
  • claim_correction_history

    • Module 10: Claim Submission

    Purpose:

    Submit claims electronically to insurance payers and track submission status.

    What User Does:

  • Opens "Claims" module
  • Filters claims by status "Ready to Submit"
  • Selects claims (single or batch)
  • Clicks "Submit Claims"
  • System shows submission confirmation

    • What Backend Does:

  • Converts claims to EDI 837 format
  • Sends to clearinghouse via Waystar API or Availity API
  • Receives submission acknowledgment (EDI 999)
  • Updates claim status to "Submitted"
  • Stores submission tracking number

    • APIs Needed:

  • Waystar API (primary clearinghouse)
  • Availity API (alternative clearinghouse)

    • API Integration Example:

    
# Submit claim via Waystar
POST https://api.waystar.com/claims/v1/submit
Headers:
  Authorization: Bearer {api_key}
  Content-Type: application/json
Body:
{
  "claims": [
    {
      "claimId": "CLM-2026-001234",
      "ediContent": "ISA*00*...",  # Full EDI 837 content
      "payerId": "00431",
      "submissionDate": "2026-05-18"
    }
  ]
}
Response:
{
  "batchId": "BATCH-20260518-001",
  "submittedClaims": 1,
  "status": "Accepted",
  "trackingNumber": "TRK-987654321"
}

    Submission Workflow:

  • Claim created → Status: "Ready"
  • Claim submitted → Status: "Submitted"
  • Acknowledgment received → Status: "Accepted" or "Rejected"
  • If rejected → Fix errors and resubmit
  • If accepted → Wait for adjudication

    • Tracking:

  • Submission date/time
  • Batch number
  • Tracking number
  • Acknowledgment status
  • Error messages (if rejected)

    • Module 11: Clearinghouse Validation

    Purpose:

    Validate, scrub, and route claims through a clearinghouse to ensure they meet payer requirements before submission to insurance companies.

    What User Does:

  • Opens "Clearinghouse Validation" module
  • Selects claims in "Ready to Submit" status
  • Initiates validation process
  • Reviews validation results (Accepted, Rejected, Accepted with warnings)
  • Opens rejected claims for correction
  • Fixes issues such as missing fields, invalid codes, and demographic mismatches
  • Resubmits corrected claims for validation
  • Tracks submission acknowledgment status

    • What Backend Does:

  • Sends claims to clearinghouse for validation
  • Performs pre-validation checks (required fields, ICD-10/CPT/NPI format validation)
  • Receives clearinghouse response (Accepted for payer submission or Rejected with error codes)
  • Stores validation results and error logs
  • Routes rejected claims to correction workflow
  • Updates claim status (Validated, Rejected, Pending Correction)
  • Generates submission acknowledgment tracking ID
  • Maintains full audit history of validation attempts
  • Triggers alerts for high rejection rates

    • APIs Needed:

  • Waystar API (primary clearinghouse validation)
  • Availity API (secondary clearinghouse integration)
  • Change Healthcare API (claims validation & routing)
  • Internal Claim Validation API (pre-scrubbing logic)
  • EDI 837/835 Processing API (claim format validation & responses)

    • Validation Rules:

  • Claim must have valid ICD-10 and CPT codes
  • Patient and provider information must be complete
  • NPI numbers must be valid and active
  • Duplicate claims are not allowed
  • Payer-specific formatting rules must be satisfied
  • Charges must match fee schedule rules
  • All required modifiers must be present where applicable

    • Data Stored:

  • clearinghouse_requests
  • clearinghouse_responses
  • claim_validation_errors
  • claim_correction_queue
  • submission_acknowledgments
  • edi_transactions

    • Module 12: Claim Status Tracking

    Purpose:

    Track and monitor the real-time status of insurance claims throughout the payer processing lifecycle using EDI 276/277 transactions, ensuring visibility into claim progress, delays, and outcomes.

    What User Does:

  • Opens "Claim Status Tracking" module
  • Searches claims using Claim ID, Patient name, Date of service, or Payer name
  • Views current claim status (Received, In Review, Pending, Approved, Denied, Paid)
  • Requests status refresh (manual inquiry)
  • Reviews claim history timeline
  • Identifies delayed or aging claims
  • Creates follow-up tasks for denied or pending claims
  • Receives automated alerts for status changes

    • What Backend Does:

  • Sends EDI 276 claim status inquiry requests to payer systems
  • Receives EDI 277 claim status responses from payers
  • Maps payer responses to internal claim status codes
  • Updates claim lifecycle status in database
  • Maintains full claim status history with timestamps
  • Triggers alerts for SLA breaches and long-pending claims
  • Generates aging analysis reports
  • Supports batch and real-time status updates
  • Logs all payer communication for audit purposes

    • APIs Needed:

  • Waystar API (primary clearinghouse for 276/277 transactions)
  • Availity API (secondary payer status tracking)
  • Change Healthcare API (claims status & EDI exchange)
  • EDI Gateway API (276/277 processing engine)
  • Internal Claim Tracking API (status mapping & history storage)

    • Tracking Rules:

  • Every claim must have a unique tracking ID
  • Status updates must be timestamped
  • Payer responses must be mapped to standardized statuses
  • Claims without updates beyond SLA threshold must trigger alerts
  • Denied claims must be routed to denial management workflow
  • Paid claims must be updated with payment details when available

    • Data Stored:

  • claim_status_tracking
  • claim_status_history
  • edi_276_requests
  • edi_277_responses
  • claim_aging_reports
  • claim_followups

    • Reports Generated:

  • Aging Report (0–30, 31–60, 61–90, 90+ days)
  • Pending Claims Report
  • Denied Claims Report
  • SLA Breach Report
  • Payer Performance Report

    • Module 13: Insurance Adjudication Tracking

    Purpose:

    Track and analyze the insurance payer adjudication process, including claim approvals, denials, partial payments, and medical necessity decisions to ensure transparency in claim processing and reimbursement outcomes.

    What User Does:

  • Opens "Adjudication Tracking" module
  • Searches claims by Claim ID, Patient name, Payer name, or Date of service
  • Views adjudication status (Approved, Denied, Partially Paid, Under Review, Pending Medical Necessity Review)
  • Opens detailed adjudication breakdown (Allowed amount, Paid amount, Patient responsibility, Denial reasons)
  • Reviews contract compliance information
  • Identifies underpaid or denied claims
  • Initiates appeal workflow if required
  • Monitors payer response timelines and delays

    • What Backend Does:

  • Receives adjudication data from payer systems via EDI 835 / payer APIs
  • Parses and stores adjudication results (Allowed amounts, Adjustments, Denial codes)
  • Maps adjudication outcomes to internal claim records
  • Tracks medical necessity review decisions
  • Validates claim payments against contract fee schedules
  • Detects underpayments and discrepancies
  • Updates claim financial status (Fully Paid, Partially Paid, Denied)
  • Maintains adjudication history and audit logs
  • Triggers alerts for denials, underpayments, and high-value claim rejections
  • Generates payer performance and turnaround analytics

    • APIs Needed:

  • Waystar API (primary adjudication & EDI 835 processing)
  • Availity API (payer response & adjudication data)
  • Change Healthcare API (claims payment & adjudication feed)
  • EDI 835 Processing API (remittance & adjudication details)
  • Contract Management API (fee schedule validation)

    • Tracking Rules:

  • Every claim must have linked adjudication record
  • All payer responses must be stored in normalized format
  • Denial codes (CARC/RARC) must be recorded
  • Payment must match allowed amount unless adjustment is applied
  • Medical necessity denials must be flagged separately
  • Contract variance must be calculated for every claim
  • Partial payments must be broken into components (Allowed amount, Paid amount, Patient responsibility)

    • Data Stored:

  • adjudication_records
  • payment_details
  • denial_codes
  • contract_variance_logs
  • medical_necessity_reviews
  • payer_processing_metrics

    • Reports Generated:

  • Adjudication Summary Report
  • Denial Reason Analysis Report
  • Underpayment Detection Report
  • Payer Turnaround Time Report
  • Contract Variance Report
  • Medical Necessity Denial Report

    • Module 14: ERA Processing

    Purpose:

    Process Electronic Remittance Advice (ERA) files (EDI 835) received from insurance payers to reconcile payments, extract denials, and automate posting of financial transactions.

    What User Does:

  • Opens "ERA Processing" module
  • Uploads or imports ERA files (EDI 835)
  • Selects payer or batch for processing
  • Initiates ERA processing workflow
  • Reviews processed results (Payments posted, Denials extracted, Adjustments applied)
  • Validates reconciliation reports
  • Manually corrects unmatched payments if needed
  • Confirms auto-posting to accounts receivable

    • What Backend Does:

  • Receives ERA (EDI 835) file from payer or clearinghouse
  • Parses 835 transaction segments (Payment details, Claim adjustments, Denial codes CARC/RARC)
  • Matches ERA data with existing claims
  • Performs payment reconciliation (claim level and line item level matching)
  • Posts payments automatically to patient accounts (if auto-post enabled)
  • Updates claim status (Paid, Partially Paid, Denied)
  • Extracts and categorizes denial reasons
  • Applies contractual adjustments based on payer rules
  • Flags unmatched or missing claims for review
  • Generates audit trail for all financial postings

    • APIs Needed:

  • Waystar API (ERA/EDI 835 ingestion & processing)
  • Availity API (ERA file exchange)
  • Change Healthcare API (remittance data processing)
  • EDI 835 Parsing Engine API
  • Accounting/AR System API (payment posting & reconciliation)
  • Internal Claim Matching API (claim-to-payment reconciliation logic)

    • Processing Rules:

  • Each ERA must be linked to a valid payer and claim batch
  • Payments must be matched using Claim number, Patient ID, and Service date
  • Denial codes (CARC/RARC) must be extracted and stored
  • Contractual adjustments must be applied automatically
  • Overpayments must be flagged for refund workflow
  • Unmatched payments must go to exception queue
  • Duplicate ERA files must be detected and ignored

    • Data Stored:

  • era_files
  • era_transactions
  • payment_postings
  • denial_records
  • adjustment_records
  • unmatched_payments
  • reconciliation_logs

    • Reports Generated:

  • ERA Processing Summary Report
  • Payment Reconciliation Report
  • Denial Summary Report
  • Unmatched Payments Report
  • Adjustment Analysis Report
  • AR Posting Audit Report

    • Module 15: Payment Posting

    Purpose:

    Record payments from insurance companies and patients, and reconcile with claims.

    What User Does:

  • Opens "Payment Posting" module
  • Uploads ERA file (EDI 835) or enters manual payment
  • System auto-matches payments to claims
  • Reviews matches
  • Posts payments
  • Handles adjustments and denials

    • What Backend Does:

  • Parses ERA file (EDI 835)
  • Extracts payment information:
  • Claim number
  • Paid amount
  • Adjustment codes
  • Patient responsibility
  • Matches payments to claims automatically
  • Updates claim status:
  • Fully paid
  • Partially paid
  • Denied
  • Calculates patient balance
  • Updates accounts receivable

    • APIs Needed:

  • Waystar API (to download ERA files)
  • Stripe API (for patient credit card payments)

    • ERA Processing Example:

    
# ERA (EDI 835) structure
Claim: CLM-2026-001234
Billed Amount: $150.00
Allowed Amount: $120.00
Paid Amount: $96.00
Patient Responsibility: $24.00 (copay)
Adjustment Codes:
  - CO-45: Charge exceeds fee schedule ($30.00)
  - PR-1: Deductible ($0.00)
  - PR-2: Coinsurance ($0.00)
  - PR-3: Copay ($24.00)

    Payment Types:

  • Insurance payments (from ERA)
  • Patient payments (cash, check, credit card)
  • Adjustments (contractual, write-offs)

    • Database Updates:

  • claim_payments table
  • patient_ledger table
  • accounts_receivable table

    • Module 16: Denial Dashboard

    Purpose:

    Track denied claims and manage denial workflow to recover lost revenue.

    What User Does:

  • Opens "Denials" dashboard
  • Sees list of denied claims with:
  • Claim number
  • Patient name
  • Denial reason
  • Denial date
  • Amount
  • Days since denial
  • Clicks on claim to see details
  • Takes action:
  • Correct and resubmit
  • Appeal denial
  • Write off (if not recoverable)

    • What Backend Does:

  • Categorizes denials:
  • Eligibility issues
  • Coding errors
  • Missing information
  • Timely filing
  • Medical necessity
  • Calculates denial rate and trends
  • Sends alerts for high-priority denials

    • APIs Needed:

  • Internal: Denial tracking logic
  • Waystar API (to receive ERA files)

    • Module 17: Secondary Insurance Billing

    Purpose:

    Handle Coordination of Benefits (COB) by billing secondary and tertiary insurance payers for remaining patient balances after primary insurance has processed the claim.

    What User Does:

  • Opens "Secondary Insurance Billing" module
  • Selects claims with primary insurance payment posted
  • Reviews remaining patient balance after primary adjudication
  • Confirms eligibility for secondary billing
  • Generates secondary claim automatically or manually
  • Submits claim to secondary payer
  • Tracks submission and response status (Submitted, Accepted, Rejected, Paid)
  • Reviews secondary ERA results
  • Reconciles final patient responsibility
  • Closes claim after full settlement

    • What Backend Does:

  • Retrieves primary insurance adjudication data (ERA/835)
  • Calculates remaining balance after primary payment
  • Applies Coordination of Benefits (COB) rules
  • Determines secondary/tertiary payer hierarchy
  • Auto-generates secondary claim with adjusted charges and primary payer details
  • Submits secondary claim via clearinghouse
  • Tracks claim status using EDI 276/277 or payer APIs
  • Processes secondary ERA (EDI 835) responses
  • Updates claim lifecycle (Primary Paid, Secondary Submitted, Fully Paid / Partially Paid)
  • Handles rejection corrections and resubmissions
  • Maintains audit trail for multi-payer transactions

    • APIs Needed:

  • Waystar API (COB processing & secondary claim submission)
  • Availity API (secondary payer claim routing)
  • Change Healthcare API (multi-payer claim handling)
  • EDI 837/835 Processing API (claim submission & payment reconciliation)
  • Eligibility & COB Verification API (payer hierarchy validation)
  • Internal Claim Management API (secondary claim generation)

    • COB Processing Rules:

  • Primary insurance must be fully processed before secondary billing
  • COB rules must define payer priority order
  • Secondary claims must include primary payer EOB/ERA details
  • Patient responsibility is recalculated after each payer response
  • Duplicate secondary submissions are not allowed
  • Tertiary billing is triggered only after secondary adjudication
  • All payer payments must reconcile to total allowed amount

  • Assigns follow-up tasks to AR staff
  • Reviews collection work queues
  • Contacts payers/patients for payment status
  • Logs follow-up actions (Calls, Emails, Appeals)
  • Identifies accounts for write-off or escalation
  • Updates account status after resolution

    • What Backend Does:

  • Calculates AR aging based on invoice/claim due dates
  • Segregates accounts into aging buckets
  • Generates prioritized work queues (High-value claims first, Older accounts prioritized, High denial probability flagged)
  • Assigns follow-up tasks automatically or manually
  • Tracks all follow-up activities (communication logs and response status)
  • Updates account status after payments or adjustments
  • Identifies potential bad debt accounts
  • Supports write-off processing workflows
  • Maintains audit trail of AR actions

    • APIs Needed:

  • Billing/Claims API – balance and claim data
  • Payment Posting API – real-time updates
  • Notification API (Twilio / SendGrid) – follow-up reminders
  • CRM/Task Management API – work queue assignment
  • Analytics/Reporting API – aging and collection insights

    • AR Rules:

  • Accounts with no response after defined period move to bad debt review
  • Write-offs require supervisor approval
  • Payment or adjustment updates must immediately reflect in AR ledger
  • Duplicate follow-up tasks are not allowed

    • Module 22: Compliance & Audit

    Purpose:

    Ensure system-wide regulatory compliance (HIPAA and healthcare standards) by tracking all user activities, maintaining audit logs, monitoring access, and generating compliance reports across the RCM system.

    What User Does:

  • Opens "Compliance & Audit" module
  • Views audit dashboards (User activity logs, System access logs, Data modification history)
  • Filters logs by User, Date range, or Module (Billing, Claims, EMR, etc.)
  • Reviews security alerts (Unauthorized access attempts, Suspicious activity, Failed login attempts)
  • Generates compliance reports (HIPAA audit reports, User activity summaries)
  • Investigates flagged events
  • Exports audit logs for regulatory review

    • What Backend Does:

  • Captures every system action in real-time audit logs
  • Tracks user login/logout events, CRUD operations, financial transactions, and PHI access
  • Monitors abnormal behavior patterns (failed logins, unauthorized access, bulk exports)
  • Generates immutable append-only audit trails
  • Encrypts and secures all sensitive logs
  • Triggers security alerts for role violations and suspicious activity
  • Applies compliance mapping for HIPAA and internal policies
  • Supports long-term log retention (6–10 years or as per policy)
  • Provides audit data for internal and external reviews

    • APIs Needed:

  • Internal Audit Logging API – system-wide event tracking
  • Security Monitoring API (SIEM tools like Splunk / Datadog)
  • Identity & Access Management API (Okta / Keycloak)
  • Notification API (Twilio / SendGrid / Email/SMS alerts)
  • Encryption/KMS API (AWS KMS / Azure Key Vault)

    • Compliance Rules:

  • All PHI access must be logged
  • Logs must be immutable (no deletion allowed)
  • Every user action must be traceable to a unique user ID
  • Failed login attempts must trigger security alerts
  • Sensitive data exports must be monitored and logged
  • Role-based access control (RBAC) must be enforced
  • Audit logs must be retained as per regulatory policy (6–10 years)
  • Unauthorized access attempts must trigger immediate alerts

    • Data Stored:

  • audit_logs
  • access_logs
  • security_events
  • user_activity_history
  • compliance_reports
  • data_access_tracking

    • Reports Generated:
